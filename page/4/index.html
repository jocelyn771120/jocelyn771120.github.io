关于Uid,Euid,Gid,Egid的理解

发表于 Sep 13 2017

关于uid,gid

uid就是文件所有者的用户ID，gid就是文件所有者的组ID。
上图中的文件夹们的所有者的用户ID是one，组ID是one。

关于euid,egid

文件一定有一个所有者，但拥有打开这个文件的权限并不仅仅只有拥有者，只要是有效的uid或者是有效的gid就可以打开这个文件。

关于suid，sgid

suid==set uid==set user id，sgid同理。这两个的作用就是设置有效的uid和gid。

栗子：

用户one试图打开limpidity用户的test.txt文件，但是权限不够，用ls -l查看发现test.txt只能被limpidity用户读写。此时uid=limpidity，gid=limpidity。


通过sgid后把权限改了，可以看到此时只要是同组的都可以访问此文件，此时egid=limpidity（那是因为我把one用户和limpidity用户都放到了limpidity组里面去，如果你换一个组的话只要是同组的仍然都是egid），所以one仍然可以访问此文件。

通过这个栗子我想说的是，文件的拥有者并不是文件的唯一访问者，只要通过设置uid和gid，其他用户还是可以访问该文件的。

Pwnable Shellshock

发表于 Sep 13 2017

一开始思路错了一直往uid，gid那里钻，后来搜索“shellshock”发现这是个漏洞（哦这是我人生第一次接触漏洞，amaziiiiiiing）。

关于Shellshock

验证此漏洞存在是源于一条命令：

env x ='() { :;}; echo vulnerable' bash -c echo "this is a test"

若出现vulnerable则表示该漏洞存在。具体跟env命令无关，而是bash在处理函数定义比如'() { :;};的环境变量赋值代码上存在设计缺陷，把函数定义后的字符，也就是{后的全部字符串都当做命令执行。只要让系统接受一个含有函数定义+任意命令的环境变量赋值则可让任意命令执行。
具体参考:http://fish.ijinshan.com/cgibincheck ；
http://blog.csdn.net/u011721501/article/details/39558303 。

shellshock

所以此题可以测试下是否存在此漏洞，结果发现，是存在的。

那么就可以利用env x='() { :;}; bash -c "cat flag"' bash -c "./shellshock"拿到flag。

That’s all。

涉及到的知识点：①scanf如果没有加”&”的话，默认会取栈上的数据当作是地址。②got表可以修改的，got表上某函数的地址上的内容可以修改成别的内容。

这题的解题思路大致是：name长度为100，name和passcode都在同一个栈上，而且passcode1的地址刚好在name的最后4个字节上面，所以可以把passcode1的地址改写成got表内某个函数地址，并且向scanf的另一个参数”%d”打入system("/bin/cat flag")的地址，从而得到flag。

login里面passcode1在栈上地址是ebp-0x10(之后都用intel语体）


可以看到welcome和login是共同用一个栈的。所以name和passcode1之间的距离是0x60，也就是96个字节，name的标准长度是100，也就是说name的最后四个字节刚好是passcode1的地址。这样就可以利用第二个知识点，把got表内任意的函数地址找出来修改此地址上的内容，这里我选择的是printf函数。

Pwnable Leg

发表于 Sep 13 2017

关于ARM

ARM与x86有所不同，有arm模式和thumb模式，这两种模式下分别有不同的指令集，arm指令长度为32位，thumb指令长度为16位。

两种模式之间切换是通过bx Rn指令。若Rn的最低位是1，切换到thumb状态；若Rn的最低位是0，切换到arm状态。

对于x86来说，pc始终指向正在执行的指令，因为只有完成万一条指令的读取执行之后才会执行下一条指令。但是，arm处理器采用的是流水线机制，处理的时候pc寄存器永远指向当前执行的指令后的第二条指令（arm下的每条指令都是4个字节，thumb下每条指令都是2个字节）。

leg

一般arm模式下是以r0存放函数的返回值的。所以key1的返回值r0=0x00008cdc+8。

可以看到0x00008d00处的指令是bx r6,此时r6的最低位是1，所以工作状态转换为thumb模式。key2的返回值r0=r3=0x00008d04+4+4。

key3的返回值r0=lr,bx lr的作用是跳转到返回地址，即返回到调用者函数，然后继续执行调用体的后续指令，所以我们跳回到main函数执行调用key3后的下一条指令，地址为0x00008d80。

所以key=0x00008cdc+8+0x00008d04+4+4+0x00008d80。

缓冲区溢出的保护机制

发表于 Sep 13 2017

上周HITB GSEC CTF结束，自己在比赛中pwn只看了一道1000levels没做出来（第一次在比赛中做pwn嘻嘻嘻嘻），后来这几天去深圳参加2017TenSec今天刚回来，找了下wp看，对于小白来说还是挺难的，之后可能会把心得放上来。

今天要更新的是缓冲区溢出的保护机制，之前下了peda插件里面自带checksec功能，比赛的时候也有观察一波

  • CANARY：栈保护
    这个选项表示栈保护功能是否开启，如果没有开启，可能就会发生缓冲区溢出等漏洞，我们就可以通过覆盖栈上返回地址让shellcode得到执行。所以当此功能没有开启的时候，要注意可能会发生缓冲区溢出。
  • NX：NO-EXECUTE
    基本原理就是让数据所在内存页标识为no execute。如果开启了之后，若发生溢出执行shellcode的话cpu会抛出异常从而使得shellcode无法执行。
  • PIE：内存空间随机化
    如果开启此保护的话就会随机化处理栈上的地址使得攻击者难以确认地址。
  • RELRO
    设置符号重定向表格为只读或是在程序启动时就解析并绑定所有动态符号，这样可以防止GOT攻击。
